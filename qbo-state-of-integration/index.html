Jason's Engineering Pass (June 2026)
✓
Most significant backend work to date — pending QA confirmation
Three of the longest-standing root causes were addressed. These are backend reliability fixes. The user-facing experience layer (Phase 2) remains unbuilt on top of them.
Fixed
Invoice duplication stopped at the source
Duplicate invoices were being re-created in QBO on every sync. Safeguards added so the same record cannot be sent twice concurrently.
Fixed
Name / email matching smarter — near-identical records no longer duplicate
Fuzzy matching improved so slight spacing or spelling differences in customer names no longer create new duplicate QBO records.
Fixed
Sync errors now logged and alerted internally
Previously, sync failures disappeared silently. Errors are now logged with visibility for engineering, making future diagnosis possible.
Fixed
Connection refresh reworked — self-recovers instead of staying broken
OAuth token refresh was running too aggressively, breaking the QBO link and forcing manual reconnects. Connection now self-recovers from temporary hiccups.
Fixed
Automatic retry on QBO rate limits and temporary slowdowns
Temporary QBO API slowdowns or rate limit hits previously caused permanent sync failures. Automatic retry logic added.
Open Gaps After Jason's Pass

Jason's pass fixes the plumbing. These gaps represent the user-facing experience layer that remains unbuilt.

Open
Users cannot see sync failure indicators on their records
Errors are now logged internally, but nothing surfaces to the tenant. A failed invoice sync is still invisible to the user.
Open
No plain-language error messages exposed to users
The 401 fix was a narrow exception. Broader sync failures still produce no actionable message for the tenant — only internal logs.
Open
No manual retry at the record level
A tenant cannot self-serve a failed sync record. CS or engineering intervention still required for individual record failures.
Open
No connection status visible in Company Settings
Users have no way to confirm whether their QBO connection is active, expired, or disconnected from any normal workflow surface.
Open
No sync history visible to users or admins
There is no way for a tenant to see what synced, when it synced, and what failed — even after the fact.
Process gap
No safe CS resync protocol documented
The documented cancellation was partially caused by a CS-initiated resync creating duplicates. A written internal procedure defining when and how CS can safely trigger a resync does not exist.
Architecture — communicate or reverse
QBO-to-DripJobs sync direction was disabled — users not informed
The inbound sync direction was turned off at some point to prevent a data problem. Date and trigger undocumented. Users expecting two-way sync have no explanation. Needs a help article update and UI note, or a reversal with safeguards.
