A complete internal reference for identifying, diagnosing, and resolving substrate rate configuration errors that cause proposal prices to decrease when coat counts increase.

Background: What's Happening and Why

Some accounts report that when they increase a substrate's coat count in the Proposal Builder, the proposal price decreases rather than increases. This is counterintuitive and erodes customer confidence in the tool.

🔒
CS Context: Root Cause

This is not a defect in DripJobs' calculation engine. The math is performing correctly against the data it receives. The root cause is incorrectly configured production rate tables — specifically, rate values that produce a decreasing cost when coat count is increased. The system currently has no built-in validation to catch these at entry time, and no runtime warning to surface them in the Proposal Builder. A development initiative is underway to address both gaps.

When communicating with customers, frame this as a configuration review — not a system error. The customer-facing help article positions this correctly. Do not share language that suggests the system has a validation gap or that development is working on a fix.

⚠️
Customer Communication Guidance

Direct customers to the public-facing help article: "Understanding Substrate Rate Configuration & Multi-Coat Pricing." That article explains the behavior in a way that empowers them to audit their own rates without implying anything is wrong with the system. Do not share this internal document or reference the development tickets.

How DripJobs Calculates Substrate Prices

Each substrate has a calculation type that determines how rate values translate into a labor cost. The two most important to understand for this issue:

Calculation TypeRate RepresentsHigher Value =Lower Value =
Man Hour(s)Time required per unitMore hours → Higher priceLess time → Lower price
Linear Foot Per Man HourProduction speed (feet per hour)Faster production → Fewer hours → Lower priceSlower production → More hours → Higher price
Square Foot Per Man HourProduction speed (sq ft per hour)Faster production → Fewer hours → Lower priceSlower production → More hours → Higher price

The critical takeaway: for speed-based types, rate and price move in opposite directions. A higher LF/hr value means faster work — and therefore a lower price. This is the mechanism behind Issue Type #1 below.

3

The Three Issue Types — Full Diagnostic Detail

Review Required Issue Type A — Speed-Based Rate Inversion (Usually an Error)

Affected calculation types: Linear Foot Per Man Hour, Square Foot Per Man Hour

Pattern: The 2-coat rate is equal to or higher than the 1-coat rate, and/or the 3-coat rate is equal to or higher than the 2-coat rate.

CoatFlagged Rate (LF/hr)What This Tells the System
1 Coat15.00Painter does 15 LF/hr at 1 coat — baseline
2 Coat22.00 ↑Painter works faster at 2 coats → fewer hours → price decreases
3 Coat28.00 ↑Painter works even faster at 3 coats → price decreases further

Mechanism: DripJobs divides the substrate's total measurement by the LF/hr rate to get total hours. If the rate increases across coats, total hours decrease — and price follows. The system is doing exactly what it's told.

Common origin: Rates entered intuitively as increasing values (correct for Man Hour(s) substrates) without recognizing the speed-based type works in reverse. Also common when a substrate is copied from a Man Hour(s) template without adjusting for the new type's logic.

⚠️
CS Nuance: Not Always an Error — Confirm Before Correcting

A later coat being faster than an earlier coat is usually a misconfiguration — but legitimate real-world exceptions exist:

  • First coat includes hidden overhead: cutting in, masking, surface absorption on porous materials — second coat skips all of this
  • Primer vs. finish coat: Primer is inherently slower (sealing, uneven absorption); finish coats are smoother and faster
  • Application method changes: Coat 1 = spray + back-roll (slow); Coat 2 = spray only (fast)
  • Color coverage dynamics: First coat = heavy effort; second coat = light uniform pass
  • Crew workflow learning curve: First coat = logistics overhead; subsequent coats = optimized routine

CS approach: Flag it, explain the mechanics, then ask — "When you increase the coat count in a proposal, does the price change look right?" If the account owner confirms the behavior is intentional and accurate for their workflow, the configuration is valid. If they're surprised by a price decrease, rates need correction. Never correct without owner confirmation.

High Impact Issue Type B — Zero Multi-Coat Rates on Non-Prep Substrates

Affected calculation type: Man Hour(s) — in any category except Prep Work

Pattern: 1-coat rate has a value; 2-coat and/or 3-coat rates are 0.00.

CoatIncorrect Rate (hrs)What This Tells the System
1 Coat0.750.75 hours labor — correct
2 Coat0.00Zero hours — this substrate contributes $0 at 2 coats
3 Coat0.00Zero hours — $0 contribution at 3 coats as well

Mechanism: When the user increases coat count in the Proposal Builder, this substrate's entire labor contribution drops to $0. If a category has many affected substrates (e.g., an entire Carpentry category), the combined price drop can be substantial — hundreds of dollars.

Common origin: Bulk account setup where only the 1-coat column was populated. Also occurs when accounts are onboarded via template import that only includes 1-coat data. These errors are invisible until a customer or CS member tries to change coat count in a proposal.

Important — Prep Work is different: Prep Work substrates (power wash, masking, caulking, scraping, etc.) are performed once regardless of coat count. Zero values for 2-coat and 3-coat are intentional and correct for Prep Work. Do not flag or modify these.

Medium Impact Issue Type C — Calculation Type Mismatch Within a Category

Pattern: One or more substrates in a category use a different calculation type than the rest of the category's substrates.

Example: A "Window Trim" category where 5 of 6 substrates use Man Hour(s), but one substrate uses Linear Foot Per Man Hour — and has inverted rates (Issue Type A).

Why it matters: Type mismatch is often the diagnostic signal that leads to an Issue Type A finding. On its own it doesn't always cause incorrect pricing, but it's a reliable indicator that rates should be reviewed. Flag it and investigate the rate values on the outlier substrate.

How to Audit an Account for These Issues

Navigation Path

  1. Company Settings → Production Rates
  2. Select the relevant Area (Interior / Exterior)
  3. Click the Substrates tab
  4. Work through each Category, clicking the edit icon on each substrate to view its full rate table

Issue Type A — Speed-Based Rate Inversion Checklist

On any substrate using Linear Foot Per Man Hour or Square Foot Per Man Hour:

  • Is Coat 2 Rate strictly less than Coat 1 Rate?
  • Is Coat 3 Rate strictly less than Coat 2 Rate?
  • If either answer is No → flag for review. Then ask the account owner: "Does the price change when you increase coat count look correct to you?"
  • If they confirm it's unintentional → correct the rates. If they confirm it's intentional → document and move on.

Issue Type B — Zero Multi-Coat Rate Checklist

On any Man Hour(s) substrate in a non-Prep Work category:

  • Is Coat 2 Rate greater than 0?
  • Is Coat 3 Rate greater than 0?
  • If either is 0 → flag for Issue Type B correction

Issue Type C — Category Type Mismatch Checklist

  • Do all substrates within the category use the same calculation type?
  • If a substrate's type differs from the rest of its category → flag for review, then apply the Type A checklist to that substrate
🔍
Where to Focus First

Accounts with large numbers of custom substrates — especially those set up via bulk entry, copied templates, or CSV import — are highest risk. Prioritize: Carpentry, Trim/Door Trim/Window Trim, and any Miscellaneous catch-all category. These are the most common locations for Issue Types A and B. Prep Work zeros are expected — skip those.

5

How to Correct Each Issue

Correcting Issue Type A — Speed-Based Rate Inversion

First: Confirm with the account owner that the inversion is unintentional (see Issue Type A nuance in Section 3). If it's confirmed as a legitimate workflow difference (primer vs. finish, spray method changes, etc.), no correction is needed — document the conversation.

If correction is needed: multi-coat rates must decrease as coats increase. More coats = slower production = lower speed value = more hours = higher price.

Coat❌ Incorrect✅ CorrectLogic
1 Coat15.00 LF/hr15.00 LF/hrBaseline
2 Coat22.00 LF/hr ↑~9.00 LF/hr ↓Approx 60% of 1-coat speed
3 Coat28.00 LF/hr ↑~6.00 LF/hr ↓Approx 40% of 1-coat speed

Alternative fix: If it's simpler for the account owner, switch the substrate to Man Hour(s) and re-enter values as time (e.g., 0.5 hrs, 1.0 hrs, 1.5 hrs). This eliminates the inversion risk entirely and is appropriate for substrates that are counted by item rather than measured in feet.

Correcting Issue Type B — Zero Multi-Coat Rates

Enter appropriate hour values for the blank/zero fields. General rule of thumb (adjust based on account owner's actual production data):

CoatExample RateMultiplier
1 Coat0.75 hrs (existing)—
2 Coat1.35 hrs~1.8× the 1-coat value
3 Coat1.95 hrs~2.6× the 1-coat value

Correcting Issue Type C — Type Mismatch

Review with account owner. Change the calculation type to match the category standard, then re-enter rates using the correct logic for that type. If the different type is intentional and confirmed, document it and apply the Type A checklist to the outlier substrate's rate values.

🚨
Always Involve the Account Owner Before Changing Rates

Rate values are business data. CS should not modify substrate rates without the account owner's explicit awareness and approval. Identify, explain, and confirm — then make the correction together or guide the owner through it themselves.

CS Support Playbook

Use this flow when a customer reports that their proposal price decreases when they add more coats, or when you suspect production rate misconfiguration during an account audit.

1
Triage — Confirm the Symptom
  1. Ask: "Which substrate or area shows the price decrease, and which coat count change triggers it?"
  2. If they can't isolate it, ask them to duplicate the affected package and change only one substrate's coat count at a time to identify the culprit.
  3. Confirm the symptom is isolated to specific substrates — not a global pricing issue.
2
Audit the Identified Substrate
  1. Navigate to the account's Production Rates (with their permission, or guide them through it).
  2. Locate the substrate they identified. Note its calculation type and all three coat rate values.
  3. Apply the Issue Type A checklist (speed-based inversion) and Issue Type B checklist (zero multi-coat) from Section 4.
3
Expand the Audit
  1. Once you find one issue, check the entire category — issues almost always appear in batches, especially for Issue Type B.
  2. Prioritize: Carpentry, Window/Door Trim, and any Miscellaneous catch-all category. Skip Prep Work zeros — those are correct.
  3. Flag every affected substrate before beginning any corrections.
4
Explain and Correct
  1. Use plain language with the account owner. Reference the public help article to frame this as a configuration review. Do not use language suggesting a system gap.
  2. Show them the specific rate values that are misconfigured and explain what they're telling the system (see Section 6 for reference tables).
  3. Let them review and approve corrected values before you or they save anything.
5
Verify the Fix
  1. After corrections are saved, return to the Proposal Builder and change the affected substrate's coat count. Confirm the price now increases as expected.
  2. Repeat for all corrected substrates.
  3. Alert the account owner that any previously sent proposals that used these substrates at 2+ coats should be reviewed for accurate pricing. Do not make changes to sent proposals — that decision belongs to the account owner.
🚨
Already-Sent Proposals

If a customer sent proposals while the misconfiguration was in place, those proposals may have been underpriced. Surface this carefully and privately. Do not modify sent proposals without the account owner's explicit review and approval. This is a revenue impact conversation — handle with care.

🔧
Development Status

Three enhancement tickets are in Active Triage to address this permanently: (1) a Proposal Builder warning when coat count increase causes a price decrease, (2) a Production Rates Health Check tool for account-wide auditing, and (3) inline validation on save in the substrate configuration form. Until these ship, manual audit is the recommended path — use this playbook.
